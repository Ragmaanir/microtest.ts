# PRD: microtest.ts

## Purpose

`microtest.ts` is a minimal TypeScript testing library inspired by `Ragmaanir/microtest`.

The original `microtest` is written in Crystal, so some features, especially macro-powered assertions, cannot be copied directly. This project should preserve the spirit of `microtest`: small API, explicit behavior, simple reporters, deterministic randomization, and self-hosted tests.

## Goals

- Provide a minimal test DSL for suites, tests, hooks, and assertions.
- Keep the core library small, dependency-free where practical, and strongly typed.
- Support multiple reporters.
- Support focused tests.
- Support suite-level and global hooks.
- Support `before`, `after`, and `around` hooks.
- Require every test to belong to a suite.
- Randomize test order by default.
- Support deterministic re-runs through `SEED`.
- Provide colored terminal output.
- Use an injected writer for reporter output so reporters are easy to test.
- Run TypeScript tests through `tsx`.
- Test the library with its own testing API.

## Non-Goals

- No nested suites.
- No DOM-specific API in the core library.
- No Jest/Vitest-compatible mocking system.
- No snapshot testing in the initial version.
- No heavy assertion dependency.
- No mandatory CLI in the initial version.
- No full power-assert implementation in the initial version.

## Runtime And Project Constraints

- TypeScript only.
- ESM only.
- pnpm only.
- Node-oriented.
- Tests should be executable with `tsx`.
- Source files should use `.js` extensions in relative imports.

## Test Discovery And Execution

The library should support running:

- One test file.
- A file pattern.
- All project tests.

The preferred initial execution model is explicit `tsx` execution rather than a mandatory CLI.

Examples:

```sh
tsx test/assert.test.ts
tsx "test/**/*.test.ts"
tsx microtest.ts
```

If pattern execution cannot be handled cleanly with `tsx` alone, the project may add a small helper entrypoint or optional CLI later.

The library should not require a CLI for basic usage. A consuming project should be able to define:

```json
{
  "scripts": {
    "test": "tsx test/index.ts"
  }
}
```

## DSL

### Recommended Test Style

```typescript
import { assert, before, after, around, suite, test } from "../src/index.js"

suite("Counter", () => {
  before(() => {
    // suite setup
  })

  after(() => {
    // suite teardown
  })

  around(async (next) => {
    // before each test
    await next()
    // after each test
  })

  test("#increment", () => {
    const counter = new Counter()

    counter.increment()

    assert(counter.value === 1)
    assert.equal(counter.value, 1)
  })
})
```

Suites should run themselves automatically. Test files should not need to call `run()` at the bottom.

The suite name should be prepended automatically in reporter output and result metadata. For example, `suite("Counter", ...)` with `test("#increment", ...)` should be reported as `Counter#increment`.

### Suite API

```typescript
suite(name: string, fn: () => void): void
```

Rules:

- Suites cannot be nested.
- Calling `suite` inside another `suite` should throw a clear error.
- Suite declaration order should remain stable.
- Tests inside each suite should be randomized by default.
- Suites should schedule themselves for automatic execution.

### Test API

```typescript
test(name: string, fn: TestFunction): void
test.only(name: string, fn: TestFunction): void
test.skip(name: string, fn?: TestFunction): void
```

```typescript
type TestFunction = () => void | Promise<void>
```

Rules:

- Every test must belong to a suite.
- Calling `test` outside a suite should throw a clear error.
- Test names should be relative to the suite name.
- Reporter output and result metadata should expose both the local test name and the full suite-prefixed name.
- `test.only` marks a focused test.
- If focused tests exist, only focused tests run.
- Non-focused tests should be reported as skipped because of focus.
- `test.skip` marks a skipped test.
- A test without a function may be treated as pending if that API is added.

## Hooks

### Suite Hooks

The hook names should be:

```typescript
before(fn: HookFunction): void
after(fn: HookFunction): void
around(fn: AroundHookFunction): void
```

```typescript
type HookFunction = () => void | Promise<void>
type AroundHookFunction = (next: () => Promise<void>) => void | Promise<void>
```

Rules:

- `before` runs once before the suite's tests.
- `after` runs once after the suite's tests.
- `around` wraps each test in the suite.
- Multiple hooks of the same type should run in declaration order.

### Global Hooks

The library should support global hooks through the `Microtest` namespace:

```typescript
Microtest.before(() => {
  // before all suites
})

Microtest.after(() => {
  // after all suites
})

Microtest.around(async (next) => {
  // around every test
  await next()
})
```

Rules:

- `before`, `after`, and `around` register suite-local hooks and must be called inside a suite.
- `Microtest.before`, `Microtest.after`, and `Microtest.around` register global hooks.
- Global `before` runs before all suites.
- Global `after` runs after all suites.
- Global `around` wraps every test.

### Hook Execution Order

For each run:

1. Global `before` hooks.
2. Suite `before` hooks.
3. For each selected test:
   - Global `around` hooks.
   - Suite `around` hooks.
   - Test body.
4. Suite `after` hooks.
5. Global `after` hooks.

Around hooks should nest in declaration order.

Example:

```typescript
Microtest.around(async (next) => {
  writer.write("global before")
  await next()
  writer.write("global after")
})

suite("Example", () => {
  around(async (next) => {
    writer.write("suite before")
    await next()
    writer.write("suite after")
  })
})
```

Expected wrapping:

```text
global before
suite before
test
suite after
global after
```

## Assertions

### General Assert

```typescript
assert(condition: unknown): asserts condition
```

Behavior:

- Passes for truthy values.
- Throws `AssertionFailure` for falsy values.
- Should include useful failure metadata where practical.

### Equality Assert

```typescript
assert.equal<T>(actual: T, expected: T): void
```

Behavior:

- Compares values using `Object.is` for primitives.
- Should support useful diff output for simple objects and arrays later.
- Throws `AssertionFailure` on mismatch.

### Throws Assert

```typescript
assert.throws(fn: () => unknown): void
assert.throws(fn: () => unknown, expected: RegExp | string | ErrorConstructor): void
```

Behavior:

- Passes when the function throws.
- Fails when the function does not throw.
- If an expected matcher is provided, the thrown error must match it.
- Throws `AssertionFailure` on assertion failure.

### AssertionFailure

The library should define a custom assertion error:

```typescript
class AssertionFailure extends Error {
  name: "AssertionFailure"
}
```

This should not be an issue. It is useful because the runner can distinguish failed assertions from unexpected runtime errors:

- `AssertionFailure` means the test failed.
- Any other thrown value means the test errored.

The implementation should still handle non-`Error` thrown values defensively.

### Power Assert Direction

The initial version should not attempt full power asserts.

Crystal can implement power asserts more naturally because of language-level macro capabilities. TypeScript cannot reliably inspect expression structure at runtime.

Initial support should be:

```typescript
assert(value > 3)
assert.equal(value, 3)
```

Optional experimental syntax later:

```typescript
assert.that(() => value > 3)
```

This could inspect `fn.toString()`, but it should not be treated as robust because bundlers, transpilers, and minifiers can rewrite function source.

True power asserts should be investigated later as a separate feature, likely requiring a transform or compiler integration.

## Reporters

Reporters should receive structured events and write through an injected writer.

### Writer API

```typescript
interface Writer {
  write(text: string): void | Promise<void>
}
```

The default writer should write to `process.stdout`.

Tests can inject an in-memory writer:

```typescript
class MemoryWriter implements Writer {
  output = ""

  write(text: string) {
    this.output += text
  }
}
```

### Reporter API

```typescript
interface Reporter {
  run_start?(event: RunStartEvent): void | Promise<void>
  suite_start?(event: SuiteStartEvent): void | Promise<void>
  test_start?(event: TestStartEvent): void | Promise<void>
  test_result?(event: TestResultEvent): void | Promise<void>
  suite_end?(event: SuiteEndEvent): void | Promise<void>
  run_end?(event: RunEndEvent): void | Promise<void>
}
```

Reporter events should include enough information for output and tests:

- Suite name.
- Test name.
- Status.
- Duration.
- Error or failure.
- Seed.
- Counts.

### Built-In Reporters

Initial built-in reporters:

- `progress_reporter`: compact progress output using configurable symbols.
- `description_reporter`: suite and test descriptions.
- `error_reporter`: failure and error details.
- `summary_reporter`: totals, duration, and seed.
- `slow_test_reporter`: slowest tests.

The progress reporter should not be limited to fixed `.`, `F`, `E`, `S` output. It should support configurable symbols similar to `microtest`.

Default symbols should be colored and readable in modern terminals, for example:

- Passed: green dot or green tick.
- Failed: red cross.
- Errored: red bang.
- Skipped or pending: gray or yellow skip marker.

Example:

```typescript
progress_reporter({
  symbols: {
    passed: "·",
    failed: "✗",
    errored: "!",
    skipped: "s",
    pending: "p",
  },
})
```

Reporter presets:

```typescript
await run({
  reporters: ["progress"]
})

await run({
  reporters: ["description"]
})
```

Expanded equivalent:

```typescript
await run({
  reporters: [
    progress_reporter(),
    error_reporter(),
    slow_test_reporter(),
    summary_reporter(),
  ],
})
```

## Result Model

```typescript
type TestStatus = "passed" | "failed" | "errored" | "skipped" | "pending"
```

Definitions:

- `passed`: test completed without throwing.
- `failed`: test threw `AssertionFailure`.
- `errored`: test threw anything else.
- `skipped`: test was explicitly skipped or skipped because focus mode is active.
- `pending`: test was declared without an implementation, if supported.

```typescript
interface TestResult {
  suite_name: string
  test_name: string
  full_name: string
  status: TestStatus
  duration_ms: number
  error?: unknown
}
```

## Randomization

Test order should be randomized by default.

Seed rules:

- Use `process.env.SEED` when provided.
- Generate a seed when `SEED` is not provided.
- Print the seed in reporter output.
- Include the seed in the run result.
- A failed randomized run must be reproducible by re-running with the same `SEED`.

Recommended v1 behavior:

- Keep suite order stable.
- Randomize test order within each suite.

Future option:

- Add full suite randomization if needed.

## Automatic Run Behavior

Suites should run themselves automatically. A test file should not need:

```typescript
await run()
```

The implementation should schedule execution after the current module has finished registering suites and tests.

Possible implementation approach:

- Register suites synchronously during module evaluation.
- Schedule a single run using `queueMicrotask`, `setImmediate`, or a similar mechanism.
- Ensure the run is scheduled only once.
- Ensure all suites in the current process are included.

The automatic run should set `process.exitCode = 1` when the final result fails.

This is required so shell chains work correctly:

```sh
pnpm test && then_do_this
```

`run()` may still exist as an internal or advanced API, but automatic execution should be the normal user-facing workflow.

## Run API

```typescript
await run()
```

```typescript
await run({
  seed: "12345",
  reporters: [
    progress_reporter(),
    error_reporter(),
    summary_reporter(),
  ],
  writer,
})
```

```typescript
interface RunOptions {
  seed?: string
  reporters?: Reporter[] | ReporterPreset[]
  writer?: Writer
  color?: boolean | "auto"
}
```

The run result should be returned programmatically:

```typescript
interface RunResult {
  seed: string
  status: "passed" | "failed"
  suites: SuiteResult[]
  tests: TestResult[]
  duration_ms: number
}
```

Rules:

- `run()` should set `process.exitCode = 1` on failure by default.
- Automatic suite execution should use the same runner behavior.
- If needed for internal tests, a future option may disable exit-code mutation.

## Colored Terminal Output

Terminal output should be colored by default when supported.

Rules:

- Disable color when `NO_COLOR` is set.
- Disable color when output is not a TTY, unless forced.
- Force color when `FORCE_COLOR=1`.
- Keep coloring implementation small.

Suggested colors:

- Passed: green.
- Failed: red.
- Errored: red.
- Skipped or pending: yellow or gray.
- Suite names: cyan or bold.
- Metadata and durations: gray.

## CLI Position

A CLI is not required for the initial version if `tsx` execution is sufficient.

Preferred initial model:

```json
{
  "scripts": {
    "test": "tsx test/index.ts"
  }
}
```

A CLI may be added later if the project needs:

- Built-in glob expansion.
- Better multi-file execution.
- Watch mode.
- Standardized reporter selection through flags.
- Standardized seed passing through flags.

Potential future CLI:

```sh
microtest test/**/*.test.ts
microtest test/assert.test.ts --seed 12345
microtest --reporter description
```

## Self-Hosted Tests

The project must test itself using `microtest.ts`.

Example:

```typescript
import { assert, suite, test } from "../src/index.js"

suite("assert", () => {
  test("assert fails for false", () => {
    assert.throws(() => assert(false))
  })

  test("assert passes for true", () => {
    assert(true)
  })
})
```

Coverage priorities:

- Assertion behavior.
- `AssertionFailure` classification.
- Suite declaration.
- Rejection of nested suites.
- Focused tests.
- Skipped tests.
- Hook order.
- Around hook wrapping.
- Global hooks.
- Reporter events.
- Injected writer output.
- Seeded randomization.

Bootstrap rule:

- During initial implementation, manual verification scripts are acceptable until enough of the library exists to test itself.
- Once the core runner and assert API exist, new tests should use `microtest.ts`.

## Implementation Milestones

### Milestone 1: Core Model And Assertions

- Define suite and test model.
- Implement `suite`.
- Implement `test`.
- Implement `AssertionFailure`.
- Implement `assert`.
- Implement `test.only`.
- Implement `test.skip`.
- Implement `assert.equal`.
- Implement `assert.throws`.

### Milestone 2: Runner

- Implement `run`.
- Implement automatic run scheduling for suites.
- Collect results.
- Distinguish failed assertions from runtime errors.
- Return structured `RunResult`.
- Support async tests.
- Set `process.exitCode = 1` when the run fails.

### Milestone 3: Hooks

- Implement namespaced global `Microtest.before`, `Microtest.after`, `Microtest.around`.
- Implement suite-local `before`, `after`, `around`.
- Enforce hook scoping rules.
- Verify hook execution order.

### Milestone 4: Randomization

- Implement deterministic seeded randomization.
- Read `SEED` from the environment.
- Generate seed when not provided.
- Include seed in result and reporter output.

### Milestone 5: Reporters

- Implement `Writer`.
- Implement reporter event dispatch.
- Implement progress reporter with configurable symbols.
- Implement description reporter.
- Implement error reporter.
- Implement summary reporter.
- Implement slow test reporter.

### Milestone 6: Self-Hosted Tests

- Add tests using `microtest.ts`.
- Add an aggregate test entrypoint.
- Add package script using `tsx`.

### Milestone 7: Optional CLI Evaluation

- Decide whether `tsx` scripts are enough.
- Add CLI only if it materially improves file/pattern/all-test execution.

## Open Questions

1. Should skipped tests caused by focus mode be printed individually, or only counted in the summary?

2. Should `test.skip(name)` without a function be allowed?

3. Should `test(name)` without a function create a pending test?

4. Should `assert.equal` use strict `Object.is` only in v1, or should it include deep equality for arrays and plain objects immediately?

5. Should the first release include a tiny glob runner, or should `test/index.ts` import all test files manually?

6. Should reporter presets be strings like `"progress"` or exported functions only?

7. Should automatic suite execution use `queueMicrotask`, `setImmediate`, or another scheduling mechanism?

8. Should `run()` expose an option such as `{ set_exit_code: false }` for testing the runner itself?

9. Should suite-prefixed names always concatenate directly, or should plain names receive a separator? Example: should `suite("Counter")` plus `test("increments")` become `Counterincrements`, `Counter increments`, or should users be expected to write `test("#increments")`?
